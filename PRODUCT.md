# DevOps Watcher — 产品需求文档

> 版本：6.0（Chrome 插件版）｜ 日期：2026-03-09
>
> 技术实现细节请参见 [README.md](./README.md)。

---

## 一、产品目标

监控阿里云效 DevOps 需求池的变化，**第一时间发现新需求进入或需求被移除，抢占先机。**

用户打开云效需求列表页面后，插件在后台自动运行，无需手动操作。当监控的需求池发生变化时，立即弹出桌面通知并播放声音，通知内容精确到具体需求名称。

---

## 二、适用页面

仅在以下 URL 激活插件功能：

```
https://devops.aliyun.com/projex/project/*
```

并且 URL 中必须包含 `?`（查询参数），否则插件静默不运行。
在存在多个云效页面时，通过浏览器右上角扩展图标的 **Popup 弹窗**中的 ON/OFF 开关控制当前 tab 是否执行监控。

## 三、核心功能

### 3.1 需求数量监控

- 实时追踪每个需求池的需求总数和具体需求
- 记录每次检测的数量快照，形成历史趋势
- 面板实时显示各需求池当前数量

### 3.2 需求变化检测

- 精确对比两次检测之间**哪些需求新增、哪些被移除**
- 检测之后对比需求数和具体需求，有变化才通知
- **首次检测该需求池时**：只保存当前列表做基准，不发通知（避免启动误报）
- **需求名比对规则**：精确字符串全匹配。若某需求的名称被修改，系统会将其识别为"旧名称移除 + 新名称新增"两条变化

### 3.3 需求详情查看

- 在面板"需求列表"区域中，按需求池分组展示所有需求
- 所有需求项均可点击，点击后实时获取该工作项的字段详情并内联展示
- 详情数据来自云效工作项字段 API（字段定义 + 字段值），通过注入脚本主动发起同源 GET 请求获取
- 自动过滤掉 `subject` 和富文本字段，仅展示有值的字段
- 已获取的详情自动缓存，再次点击无需重复请求
- 获取失败时显示错误信息和重试按钮

### 3.4 变化记录持久化

- 每次检测到的变化（新增/移除了哪些需求）会持久化到本地数据库
- 历史变化记录累积保存，最多保留 **200 条**
- 每条变化记录包含：需求池名称、变化内容、检测时间戳
- 可按需求池单独清除变化记录
- 关闭页面后重新打开，历史变化记录自动恢复

### 3.5 桌面通知

触发条件：需求数量发生变化 **或** 需求列表发生变化（二者满足其一即通知）。

**通知标题格式：**

```
📢 【需求池名称】 [+新增数量 -移除数量]
```

示例：`📢 【iOS 需求池】 [+2 -1]`

若只有数量变化、没有需求名变化，则不显示 `[+N -M]` 部分。

**通知正文格式：**

```
📈 旧数量→新数量 (+变化量)        ← 数量增加时
📉 旧数量→新数量 (-变化量)        ← 数量减少时
➕ 新增:
  • 需求名称一
  • 需求名称二（超过 30 字自动截断并加省略号）
➖ 移除:
  • 旧需求名称一
```

通知自动消失时间：**10 秒**。

### 3.6 声音提醒

- 有变化触发通知时同步播放提示音
- 使用 **Web Audio API** 程序化生成提示音（正弦波短促蜂鸣），无需外部音频文件

### 3.7 自动循环检测

- 插件启动后自动开始循环检测，无需用户手动触发
- 每轮检测间隔随机，范围 **60 ～ 120 秒**
- 多个需求池的检测操作**均匀分散**在整个倒计时周期内，不集中在同一时刻
- 倒计时期间，面板上显示实时倒计时数字
- 页面隐藏（切换到其他标签页）时**继续运行所有功能**，不暂停

## 四、检测逻辑

### 4.1 工作方式

**监控流程**不直接发起额外网络请求，而是**监听并复用页面本身已有的 API 请求数据**，做到零额外网络负担。

当云效页面自行请求需求列表接口时，插件从响应数据中提取：

- 需求总数
- 所有需求的列表（含名称 `subject` 和唯一标识 `identifier`）

**需求详情**是唯一的例外：当用户在面板中主动点击某条需求时，插件通过注入脚本发起同源 GET 请求获取该工作项的字段定义和字段值。此请求仅在用户交互时按需触发，不在自动监控流程中执行。

### 4.2 多需求池检测流程

以配置了"iOS 需求池"和"Android 需求池"两个监控目标为例，每轮检测的完整步骤：

1. 在侧边栏菜单中找到"iOS 需求池"对应的菜单项
2. 模拟点击该菜单项
3. 等待页面数据加载完成
4. 从 API 数据中提取需求总数和需求列表
5. 若有多页，自动翻页收集所有分页数据
6. 与上次保存的数据比对，检测变化
7. 若有变化，立即发送通知
8. 按时间间隔分散，对"Android 需求池"重复步骤 2～7
9. 两个需求池处理完毕后，等待下一轮倒计时

**操作时间分布：** 所有操作（含翻页）在倒计时周期内均匀分布，首尾留出缓冲时间。例如 90 秒周期内 4 个操作，分别在约 18s、36s、54s、72s 执行。

**首次启动优化：** 如果用户打开页面时已在某个目标需求池上，直接读取当前页面数据，首轮调度跳过该池避免重复点击。首次收集到的数据仅作为初始基准保存，**不触发变化通知**。

### 4.3 分页处理

需求池中的需求数量可能超过单页显示上限，插件通过**模拟点击翻页按钮**触发后续分页数据加载：

- 翻页操作与池切换操作统一纳入时间分布，确保整体节奏均匀
- 每翻一页后等待内容加载就绪
- 最多处理 **10 页**
- 合并所有页的需求后**去重**再进行比对
- **分页状态回退**：切换需求池后如果检测到分页不在第 1 页，自动点击分页控件回退到第 1 页后重新获取数据
- **数据完整性校验**：翻页收集完成后，若实际收集数量与 `totalCount` 不一致，跳过本轮变化检测，保留上次快照作为基准，避免因不完整数据产生虚假变化

### 4.4 内容加载等待

点击菜单项后，插件按三段策略依次等待页面数据就绪：

**第一段：API 数据快速判断**
若拦截到的 API 数据距当前时刻不超过 **2 秒**，视为内容已就绪，立即进入下一步。

**第二段：快速轮询**
以 **50ms 间隔最多轮询 10 次**（约 500ms），检查页面加载状态。

**第三段：DOM 变化监听**
快速轮询后仍未就绪，监听页面变化直至总超时（**10 秒**）。

**超时处理：** 超时后自动重试（最多 3 次），全部失败则跳过该池，不影响后续检测。

## 五、UI 面板

### 5.1 悬浮球（收起状态）

- 默认收起为一个小圆形悬浮球（直径 52px），停留在页面右下角
- 球上显示**倒计时进度环**和倒计时数字
    - 倒计时充足时：白色
    - 剩余不足 30 秒：橙色
    - 剩余不足 10 秒：红色
- 点击悬浮球展开完整面板
- 有通知触发时，悬浮球闪烁蓝色光晕 3 秒

### 5.2 完整面板（展开状态）

点击悬浮球后展开为 **宽 360px 的悬浮面板**，最大高度为视口的 85%，内容可滚动。整体采用**浅色主题**。滚动区域使用 `scrollbar-gutter: stable` 预留滚动条空间，防止内容溢出状态变化时面板宽度跳动。

面板组成（从上到下）：

#### ① 顶部标题栏

- 左侧：插件名称 "DevOps Watcher" + 监控目标数量角标
- 右侧：当前倒计时秒数 + 关闭按钮（收起为悬浮球）
- 整个标题栏作为拖拽把手

#### ② 需求池数量卡片

- 每个监控目标显示一张卡片
- 卡片显示：目标名称 + 当前数量（数字，颜色区分不同目标）
- 目标数 ≤ 2 时按数量排列，超过 2 个时固定 2 列

#### ③ 状态栏

- 左侧：当前运行状态文字（下一步操作、等待倒计时等）
    - 正常状态：绿色
    - 异常状态：橙色
- 右侧：当前 JS 内存占用
    - 正常（< 50%）：绿色
    - 偏高（50%～80%）：橙色
    - 危险（> 80%）：红色

#### ④ 需求列表（可折叠）

- 按需求池分组展示所有需求，每个池显示需求数量角标
- 每条需求带序号、名称和展开箭头，**所有需求项均可点击**
- 点击需求项后展开详情区域，实时请求该工作项的字段信息：
    - 加载中：显示旋转加载动画
    - 加载成功：以标签-值对的形式展示所有有值字段（过滤掉 `subject` 和富文本）
    - 加载失败：显示错误信息 + 重试按钮
    - 无 `identifier`：显示"暂无详情数据"
- 再次点击已展开的需求项可收起详情
- 详情数据自动缓存，展开已缓存的需求无需重新请求

#### ⑤ 趋势图表（可折叠）

- 使用 **Chart.js** 折线图展示各需求池数量的历史趋势
- 多目标用不同颜色的折线区分
- 最多显示最近 100 条历史数据点

#### ⑥ 需求变化（可折叠）

- 按需求池分组，每个池显示变化条数角标
- 每条变化记录显示检测时间（MM/DD HH:mm:ss 格式）和数量差值（如 "197 → 198 (+1)"）
- 新增需求：绿色区块标题 "+ 新增" + 数量 + 需求名完整列表
- 移除需求：红色区块标题 "- 移除" + 数量 + 需求名完整列表
- **需求名可选择复制**（`user-select: text`），方便用户引用
- 每个需求池右侧有"清除"按钮，可单独清除该池的变化记录
- 变化记录持久化保存，最多累积 200 条
- 无变化时显示"暂无变化"

#### ⑦ 历史记录（可折叠）

- 按时间倒序列出每次检测的快照
- 每条记录显示：时间（MM/DD HH:mm:ss）+ 各需求池数量 + 与上次的差值（+N 绿色 / -N 红色）
- 滚动到底部自动加载更多（每次加载 15 条）
- 角标显示历史总条数

#### 折叠区域互斥（手风琴模式）

上述 ④⑤⑥⑦ 四个区域采用**互斥展开**模式：展开任一区域时，其他已展开的区域自动收起。避免同时展开时面板过长导致的嵌套滚动问题。

### 5.3 位置与拖拽

- 悬浮球和面板均可自由拖拽
- 以元素中心为锚点，中心点不超出视口边界
- 点击图表、按钮、滚动区域时不触发拖拽
- **展开状态**和**收起状态**的位置分别记忆，切换时自动恢复对应的上次位置
- 位置持久化到本地数据库，刷新页面后恢复
- 默认初始位置：右下角

### 5.4 Popup 弹窗

浏览器右上角扩展图标点击后的弹窗，包含以下功能：

| 按钮 | 功能 |
|---|---|
| ON / OFF 开关 | 控制当前标签页是否执行监控 |
| 开始测试 | 关闭当前标签页，新开标签页启动全流程诊断测试，完成后自动关闭 |
| 下载日志 | 导出当前已收集的日志文件 |
| 重置位置 | 将悬浮球和面板位置恢复为默认位置 |

## 六、反检测设计

为避免被服务器识别为自动化脚本，插件从用户行为角度进行模拟：

| 策略 | 说明 |
|---|---|
| **随机间隔** | 每轮检测间隔为 60～120 秒的随机值，而非固定频率 |
| **操作均匀分散** | 所有操作（含翻页）均匀分布在倒计时周期内，首尾留缓冲 |
| **渲染等待** | 点击后短暂等待（150ms），模拟人眼阅读页面的习惯 |
| **后台持续运行** | 切走标签页时继续运行所有功能，不暂停 |
| **零额外请求（监控）** | 监控流程复用页面自身的接口请求数据，不额外发起网络请求。需求详情仅在用户主动点击时按需请求 |

## 七、自动恢复机制

### 7.1 内存超限保护

- 持续监测 JS 内存使用量
- 超过 **1024MB** 时弹出通知提示并记录日志，标记为**待刷新**
- **延迟刷新**：不立即刷新页面，而是等当前轮次的所有检测操作完成后再执行刷新，避免中途中断丢失数据
- 刷新后插件自动恢复监控

### 7.2 API 超时熔断

- 每 10 秒检测一次 API 请求状态
- 若 API 请求超过 **60 秒**未完成，弹出通知提示并记录日志，标记为**待刷新**
- **延迟刷新**：与内存超限一致，等当前轮次结束后再刷新
- 超时检测使用独立的 `lastRealResponseTime` 时间戳，不受 `invalidateFreshness()` 干扰

### 7.3 扩展上下文失效检测

- 每次检测周期同步检查 `chrome.runtime.id` 是否可用
- 扩展被重新加载或更新导致上下文失效时，记录日志并**立即刷新**页面（此时扩展已无法正常工作，不适用延迟刷新）
- 通知发送前预检上下文有效性，避免抛出 `Extension context invalidated` 错误

### 7.4 刷新方式

- 使用 `window.location.href` 赋值（附带 `_dw=timestamp` 参数）替代 `location.reload()`
- 正确处理云效 hash 路由中 `?` 出现在 `#` 之后的特殊情况
- 强制生成新 URL 导航，避免 SPA 忽略刷新

### 7.5 刷新后自动恢复

所有自动刷新操作后，插件自动重新初始化，从本地数据库恢复历史数据和面板位置，监控自动恢复。

## 八、数据持久化

| 数据类型 | 说明 | 保留上限 |
|---|---|---|
| **需求数量历史** | 每次检测的时间（MM/DD HH:mm:ss）+ 各需求池数量快照 | 500 条 |
| **需求列表快照** | 每个需求池上次检测到的所有需求名 + identifier（用于比对和详情获取） | 每池 1 条 |
| **需求变化记录** | 每次变化的详情：新增/移除了哪些需求 + 时间戳 | 200 条 |
| **面板位置** | 收起状态和展开状态各一组坐标 | 2 条 |
| **运行日志** | 每条日志含毫秒级时间戳（`ts`），刷新后恢复，支持跨刷新的精确 Duration 计算 | 2000 条 |

所有数据持久化存储在本地数据库中，关闭再打开页面后自动恢复，无需重新积累数据。

> **首次使用说明：** 首次检测到的数据仅作为初始基准写入，不触发变化通知。后续每次检测才与上次做对比。

## 九、监控目标配置

默认监控以下需求池（名称与云效侧边栏菜单的文字完全匹配）：

- `需求池`
- `iOS 需求池`
- `Android 需求池`

可根据实际项目修改为任意需求池名称，支持同时监控多个目标。增减监控目标后，界面布局（卡片列数、趋势图折线）自动适配。

## 十、云效页面适配参数

> ⚠️ 以下参数与阿里云效的页面结构强绑定，**如果云效更新改版，这些参数可能需要同步更新**。

### 10.1 DOM 选择器

| 用途 | 选择器 |
|---|---|
| 侧边栏菜单项（用于找到需求池入口并点击） | `.teamix-cloud-sidebar-side-filter-menu-item` |
| 当前激活的需求池标题（用于识别当前在哪个需求池） | `.workitemList--workitemCategory` |
| 分页"下一页"按钮（用于模拟翻页操作） | `.next-btn.next-medium.next-btn-normal.next-pagination-item.next-next` |
| 分页"上一页"按钮（用于分页状态回退） | `.next-btn.next-medium.next-btn-normal.next-pagination-item.next-prev` |

### 10.2 拦截的 API 接口

| 接口 | 路径 | 说明 |
|---|---|---|
| 需求列表 | `/projex/api/workitem/workitem/list` | 排除子路径（如 `/list/count`） |
| 字段定义 | `/projex/api/workitem/workitem/field/{workitemId}` | 工作项字段元信息 |
| 字段值 | `/projex/api/workitem/workitem/field/value/{workitemId}` | 工作项字段值 |

### 10.3 API 响应字段说明

**列表 API** — 拦截到响应后，只处理成功响应，提取以下字段：

| 字段 | 说明 |
|---|---|
| `totalCount` | 该需求池的需求总数 |
| `result[]` | 当前页的需求列表数组 |
| `result[].subject` | 每条需求的名称（用于变化检测和通知内容） |
| `result[].identifier` | 每条需求的唯一标识（用于获取工作项详情） |
| `toPage` | 当前页码（默认为 1，用于分页状态回退检测） |
| `pageSize` | 每页条数（默认为 100，用于计算总页数） |

**字段定义 / 字段值 API** — 点击需求项时主动请求，用于展示工作项详情字段。

## 十一、全量配置参数

所有可配置参数及其默认值：

### 监控核心

| 参数 | 默认值 | 说明 |

| 监控目标 | `["需求池", "iOS 需求池", "Android 需求池"]` | 需求池名称列表 |

### 分页

| 参数 | 默认值 | 说明 |
|---|---|---|
| 最大分页数 | `10` 页 | 0 表示不限制 |

### 倒计时

## 十二、版本管理

每次执行 `npm run build` 时，构建脚本自动将 `manifest.json` 和 `package.json` 中的 patch 版本号递增（如 `1.0.8` → `1.0.9`）。版本号在测试日志和 Popup 中可见，便于追踪问题发生的具体版本。

## 十三、测试模式

通过 Popup 弹窗中的「开始测试」按钮启动，会新开标签页进行全流程诊断。

- 使用**真实数据和真实交互**，不使用模拟数据
- 测试与正式监控**互斥**，不可同时运行
- 测试完成后自动下载日志文件，然后**自动关闭测试标签页**
- 可随时通过 Popup「下载日志」按钮导出已收集的日志
- 测试结束输出**结构化 Summary**，按 Phase 汇总 PASS/FAIL/WARN 计数并列出所有问题详情

测试覆盖范围（10 个 Phase）：

| Phase | 覆盖范围 |
|---|---|
| 环境检测 | URL、UA、版本号、CONFIG、Shadow DOM、inject.js 注入 |
| API 拦截 | API 拦截功能 + 数据结构校验（字段完整性，含 identifier） |
| 菜单扫描 | 侧边栏菜单扫描 |
| 数据收集 | 初始收集 + 多页感知 + 分页残留检测 |
| 倒计时调度 | 完整轮询 + 快照完整性/去重校验 + items/identifier 完整性 + 误报模式识别 |
| UI 完整性 | 悬浮球/面板/卡片/需求列表（池分组+可点击性+序号+箭头）/变化区块/文字可选择性/scrollbar-gutter 稳定性 |
| 通知功能 | 上下文预检 + 桌面通知 + 蜂鸣音 + 权限状态 |
| 数据持久化 | 快照去重/历史排序/时间间隔/重复检测/位置合理性/日志持久化 + `ts` 字段校验/快照 items+identifier 完整性 |
| 内存状态 | JS 堆内存 + 80% 阈值预警 |
| 运行时健康 | 扩展上下文/API Bridge/Recovery/DOM 选择器/分页状态/Store 状态/延迟刷新状态/日志 `ts` 完整性 |
